# RPA脚本架构和版本系统分析报告

## 1. RPA脚本组织结构

### 目录结构
- **主要位置**: `resources/rpa-scripts/` (ASAR打包的脚本)
- **缓存位置**: `userData/rpa-scripts/` (从GitHub下载的更新)
- **回退机制**: 二级系统 - 用户缓存优先，然后是打包的脚本

### 平台命名约定
系统使用中文平台名称和英文文件键名之间的映射：
- 小红书 (Xiaohongshu) → `xiaohongshu`
- 抖音 (Douyin) → `douyin`
- 快手 (Kuaishou) → `kuaishou`
- B站 (Bilibili) → `bilibili`
- 百家号 (Baijiahao) → `baijiahao`
- 微信视频号 (WeChat Channels) → `wechat-channels`

### 脚本文件格式和结构
- **主要格式**: `.mjs` 文件 (ES模块)
- **元数据**: 每个脚本导出一个 `meta` 对象，包含：
  - `platform`: 中文平台名称
  - `version`: 数字版本标识符
  - `minAppVersion`: 最低要求的应用程序版本
- **执行入口**: 每个脚本导出一个 `execute(api)` 函数，接受一个包含交互方法的API对象

## 2. 版本系统

### 版本跟踪和比较
- **基于清单**: 版本信息存储在 `manifest.json` 文件中
- **比较逻辑**: 简单的数字比较 (远程版本 > 本地版本 = 需要更新)
- **精细控制**: 每个平台可以有独立的版本号

### 版本元数据存储
- **打包清单**: `resources/rpa-scripts/manifest.json` (有限，只有小红书v23)
- **缓存清单**: `userData/rpa-scripts/manifest.json` (首次更新检查后创建)
- **脚本头部**: 每个 `.mjs` 文件在 `meta` 导出中包含嵌入的版本元数据

### 版本冲突解决
- **优先级顺序**: 缓存脚本 > 打包脚本
- **无显式冲突解决**: 系统简单地加载最高优先级的可用脚本
- **模块缓存清除**: 更新后清除注册表以确保新鲜加载

## 3. 加载和更新机制

### 脚本加载优先级/顺序
1. **第一层**: 用户缓存目录 (`userData/rpa-scripts/`) - 从GitHub下载的最新版本
2. **第二层**: 打包目录 (`resources/rpa-scripts/`) - 随应用一起发布的出厂版本

### 更新检查过程
- **后台非阻塞**: 更新在初始化后异步检查
- **GitHub集成**: 从GitHub仓库获取 (支持公共和私有仓库)
- **清单差异**: 比较远程清单与本地清单以识别需要的更新
- **选择性下载**: 仅下载有新版本的脚本
- **SHA256验证**: 根据清单中的校验和验证下载的脚本

### 更新失败时的回退机制
1. **脚本级回退**: 如果缓存脚本失败，回退到打包版本
2. **错误处理**: 优雅地处理网络故障、校验和不匹配和解析错误
3. **无硬性失败**: 即使更新失败，系统也会继续使用可用的脚本
4. **注册表清除**: 使缓存的模块无效，以确保下次尝试时新鲜加载
